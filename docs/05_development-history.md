# 개발 과정 (Development History)

## 1. 개요

Email Draft Generator는 처음부터 현재와 같은 구조를 목표로 개발한 프로젝트가 아닙니다.

실제 포워딩 업무를 수행하면서 반복적으로 작성하는 이메일의 비효율성과 담당자별 문구 차이를 개선하기 위해 작은 내부 도구로 시작했으며, 실제 사용 과정에서 발견한 문제를 지속적으로 개선하면서 현재의 구조로 발전했습니다.

이 문서는 프로젝트가 어떤 과정을 거쳐 현재 구조에 이르렀는지와, 각 단계에서 어떤 문제를 해결하기 위해 설계를 변경했는지를 정리합니다.

---

# 2. 프로젝트 시작

## 2.1 반복적인 이메일 작성

프로젝트를 시작하게 된 가장 큰 이유는 동일한 유형의 이메일를 하루에도 여러 번 반복 작성해야 하는 업무 환경이었습니다.

대표적으로 다음과 같은 이메일를 자주 작성했습니다.

- PRE-ALERT
- Customs Completed
- Transportation Invoice
- Bond Renewal
- DOA Approval

각 이메일는 목적은 다르지만 일정 부분 동일한 구조를 가지고 있었습니다.

예를 들어,

- 인사말
- 운송 일정
- 고객 안내
- 마무리 문구

등은 거의 동일한 내용을 반복해서 작성하고 있었습니다.

---

## 2.2 기존 업무 방식

기존에는 Outlook에서 이전 이메일를 복사한 뒤 필요한 부분만 수정하는 방식으로 업무를 수행했습니다.

```text
이전 이메일 검색

↓

복사

↓

내용 수정

↓

발송
```

이 방식은 간단해 보이지만 여러 문제가 존재했습니다.

- 이전 이메일를 찾는 시간이 필요함
- 고객마다 문구가 조금씩 달라짐
- 불필요한 내용이 남아 있는 경우 발생
- 필요한 안내를 누락할 가능성 존재
- 동일한 내용을 반복 작성해야 함

---

## 2.3 개선 목표

프로젝트의 첫 번째 목표는 복잡한 자동화가 아니라 **반복 입력을 줄이는 것**이었습니다.

초기 목표는 다음과 같았습니다.

- 동일한 이메일를 빠르게 작성
- 담당자 간 문구 통일
- 필요한 정보만 입력
- Outlook에서 바로 사용할 수 있는 형태 제공

---

# 3. 첫 번째 버전

## 3.1 초기 구현 범위

초기 버전은 하나의 이메일 유형만 지원하는 매우 단순한 구조였습니다.

```text
입력

↓

이메일 생성

↓

복사
```

사용자는 몇 개의 입력값만 작성하면 이메일를 생성할 수 있었습니다.

이 단계에서는 다음 기능만 구현했습니다.

- 입력 필드
- 이메일 본문 생성
- 복사 기능

---

## 3.2 첫 번째 문제

실제 사용하면서 다음과 같은 문제가 발생했습니다.

동일한 이메일라도 고객 상황에 따라 필요한 안내가 달랐습니다.

예를 들어 PRE-ALERT에서는

- CARM

- POA

- HS Code

등이 고객마다 달라졌습니다.

초기 버전에서는 모든 안내를 하나의 템플릿 안에 넣었기 때문에,

필요 없는 문단까지 함께 생성되는 문제가 있었습니다.

---

# 4. 템플릿 구조 개선

## 4.1 조건부 문단 도입

이 문제를 해결하기 위해 조건부 문단 구조를 도입했습니다.

```text
기본 템플릿

+

선택 문단

=

최종 이메일
```

사용자는 필요한 안내 항목만 선택할 수 있도록 변경했습니다.

이 변경으로

- 이메일 길이 감소
- 고객 맞춤 안내 가능
- 불필요한 문단 제거

라는 효과를 얻을 수 있었습니다.

---

## 4.2 체크박스 기반 구성

조건부 문단은 체크박스를 이용하도록 변경했습니다.

예를 들어

```text
☑ CARM

☑ POA

☐ HS Code
```

라면

CARM과 POA 안내만 이메일에 포함됩니다.

이 방식은 새로운 안내 항목을 추가하기도 쉽도록 만들어 주었습니다.

---

# 5. 이메일 유형 확장

초기에는 하나의 이메일만 지원했지만,

실제 업무에서 반복적으로 사용하는 다른 이메일들도 같은 방식으로 생성할 수 있다는 점을 확인했습니다.

이에 따라 프로젝트 범위를 점차 확장했습니다.

지원 범위는 다음과 같이 확대되었습니다.

```text
PRE-ALERT

↓

Customs Completed

↓

Transportation Invoice

↓

Bond Renewal

↓

DOA Approval
```

각 이메일는 목적과 입력값은 달랐지만,

동일한 사용자 경험을 유지하도록 설계했습니다.

---

## 5.1 공통 구조 적용

새로운 이메일를 추가하면서 모든 템플릿에 공통 구조를 적용했습니다.

```text
입력

↓

옵션 선택

↓

Preview

↓

복사
```

이 구조를 유지함으로써 사용자는 새로운 이메일 유형을 사용하더라도 별도의 학습 없이 동일한 방식으로 사용할 수 있게 되었습니다.

---

# 6. 실시간 Preview 도입

초기 버전에서는 입력을 완료한 뒤 별도의 생성 버튼을 눌러야 했습니다.

실제 사용 과정에서는 입력값을 수정하면서 결과를 계속 확인하는 경우가 많았기 때문에, 생성 버튼을 반복해서 누르는 과정이 오히려 불편했습니다.

이를 개선하기 위해 입력 즉시 Preview가 갱신되는 구조로 변경했습니다.

```text
입력

↓

즉시 렌더링

↓

Preview 갱신
```

이 변경으로 이메일를 여러 번 다시 생성하는 과정이 사라졌으며, 입력과 결과를 동시에 확인할 수 있게 되었습니다.

---

# 7. 국문 및 영문 지원

실무에서는 동일한 운송 정보를 국문 고객과 영문 고객 모두에게 안내하는 경우가 자주 발생했습니다.

초기에는 언어별로 각각 다시 입력해야 했기 때문에 같은 정보를 반복 작성하는 문제가 있었습니다.

이를 해결하기 위해 동일한 입력 데이터를 공유하는 구조를 적용했습니다.

```text
입력

↓

국문 Renderer

↓

국문 이메일


동일한 입력

↓

영문 Renderer

↓

영문 이메일
```

사용자는 한 번만 입력하면 국문과 영문 이메일를 각각 생성할 수 있게 되었으며, 반복 입력을 크게 줄일 수 있었습니다.

---

# 8. Outlook 중심의 사용 방식

프로젝트를 개발하면서 가장 중요하게 고려한 사용 환경은 Outlook이었습니다.

이메일가 생성되는 것만으로는 실제 업무에 도움이 되지 않았기 때문에,

생성된 결과를 Outlook에서 바로 사용할 수 있는 형태로 제공하는 것을 목표로 했습니다.

초기에는 일반 텍스트 복사를 사용했지만,

Outlook에 붙여넣는 과정에서

- 볼드 서식
- 밑줄
- 줄바꿈
- 문단

등이 유지되지 않는 문제가 발생했습니다.

이를 해결하기 위해 HTML 형식 복사를 지원하도록 구조를 변경했습니다.

이 변경으로 사용자는 생성된 이메일를 별도의 수정 없이 Outlook에 붙여넣어 사용할 수 있게 되었습니다.

---

# 9. 사용자 경험 개선

프로젝트를 실제 업무에 사용하면서 단순히 이메일를 생성하는 것보다, 입력 과정 자체를 단순하게 만드는 것이 생산성 향상에 더 큰 영향을 준다는 점을 확인했습니다.

이에 따라 사용자의 작업 흐름을 중심으로 여러 개선을 적용했습니다.

주요 개선 사항은 다음과 같습니다.

- 입력과 결과를 동시에 확인할 수 있는 2-Panel 레이아웃
- 이메일 유형별 탭 구성
- 국문 및 영문 전환 기능
- 입력값을 유지한 상태에서 언어 변경
- 초기화 버튼 제공
- Outlook 복사 버튼 제공

이러한 개선을 통해 사용자는 별도의 사용 방법을 익히지 않고도 반복 업무를 빠르게 수행할 수 있도록 구성했습니다.

---

## 9.1 UI 개선 방향

UI를 설계하면서 가장 중요하게 고려한 원칙은 **입력보다 확인이 쉬운 화면**이었습니다.

이메일 작성 업무에서는 입력보다 생성된 내용을 검토하는 시간이 더 중요하기 때문에, 입력 영역과 Preview를 항상 동시에 볼 수 있도록 구성했습니다.

```text
입력

↓

실시간 Preview

↓

Outlook 복사
```

이 구조는 사용자가 입력과 결과를 반복해서 비교해야 하는 과정을 줄여주었습니다.

---

# 10. 코드 구조 개선

프로젝트가 확장되면서 템플릿 수가 증가했고, 단순한 문자열 연결 방식으로는 유지보수가 어려워졌습니다.

이에 따라 공통 기능을 분리하고 역할별로 코드를 정리하는 방향으로 구조를 개선했습니다.

대표적인 개선 사항은 다음과 같습니다.

- 공통 Utility 함수 분리
- 이메일 유형별 Renderer 분리
- 날짜 변환 함수 분리
- 공통 상태 관리 구조 적용
- 중복되는 문자열 처리 제거

이러한 변경을 통해 새로운 이메일 유형을 추가하거나 기존 템플릿을 수정할 때 다른 기능에 영향을 주지 않도록 구성했습니다.

---

## 10.1 공통 구조 적용

모든 이메일 템플릿은 동일한 처리 순서를 따르도록 구성했습니다.

```text
입력

↓

상태 갱신

↓

Template Renderer

↓

Preview

↓

Clipboard
```

공통 처리 구조를 유지함으로써 이메일 유형이 증가하더라도 동일한 패턴으로 개발과 유지보수가 가능하도록 설계했습니다.

---

# 11. 자동화 확장 고려

프로젝트를 개발하면서 단순한 이메일 생성 도구에 머무르지 않고, 향후 자동화 시스템과 연결될 수 있는 구조를 지속적으로 고려했습니다.

특히 동일한 시기에 개발한 **Progress Sheets 자동화 프로젝트**와의 연계를 중요한 확장 방향으로 설정했습니다.

Progress Sheets에서는 운송 진행 상황이 지속적으로 관리되며,

- House BL#
- ETA
- Warehouse ETA
- Delivery Status
- Bond Expiry
- Contact Information

등의 데이터가 축적됩니다.

현재는 사용자가 이러한 정보를 직접 입력하지만, 향후에는 동일한 데이터를 Progress Sheets에서 자동으로 전달받을 수 있도록 설계를 고려했습니다.

---

## 11.1 Outlook Draft 자동화

최종적으로는 다음과 같은 흐름을 목표로 하고 있습니다.

```text
Progress Sheets

↓

Shipment Milestone

↓

Template Engine

↓

Outlook Draft 생성

↓

담당자 검토

↓

발송
```

이 과정에서 현재 프로젝트의 Template Engine은 이메일 생성 모듈로 그대로 재사용할 수 있습니다.

즉, 현재 프로젝트는 독립적인 이메일 생성 도구이면서 동시에 향후 자동화 워크플로우의 핵심 구성 요소를 목표로 설계했습니다.

---

# 12. 개발 과정에서 얻은 인사이트

프로젝트를 개발하면서 가장 크게 느낀 점은 **업무를 자동화하기 전에 먼저 업무를 표준화해야 한다는 것**이었습니다.

처음에는 반복 입력을 줄이는 것이 목표였지만, 개발을 진행할수록 템플릿 구조를 표준화하는 과정이 더 중요하다는 점을 확인했습니다.

특히 다음과 같은 부분이 중요했습니다.

- 담당자마다 다른 표현 통일
- 고객에게 전달하는 안내 순서 표준화
- 필수 안내 항목 누락 방지
- 반복되는 입력 최소화
- 동일한 사용자 경험 유지

이러한 표준화 과정을 거친 이후에야 자동화와의 연계가 자연스럽게 가능하다는 점을 확인할 수 있었습니다.

---

## 12.1 사용자 관점의 설계

개발 과정에서 항상 다음 질문을 기준으로 기능을 검토했습니다.

- 입력해야 하는 정보가 너무 많지는 않은가?
- 사용자가 다음 동작을 쉽게 예상할 수 있는가?
- 동일한 작업을 반복하지 않아도 되는가?
- 결과를 바로 확인할 수 있는가?
- Outlook에서 추가 수정 없이 사용할 수 있는가?

이러한 기준은 기능 추가보다 실제 업무 효율을 높이는 방향으로 프로젝트를 발전시키는 데 도움이 되었습니다.

---

# 13. 현재 프로젝트의 의의

현재 Email Draft Generator는 단순히 이메일를 생성하는 도구가 아니라, 반복적인 고객 커뮤니케이션을 표준화하기 위한 내부 업무 도구로 활용할 수 있는 구조를 갖추고 있습니다.

이 프로젝트를 통해 다음과 같은 효과를 기대할 수 있습니다.

- 반복 업무 시간 단축
- 담당자 간 이메일 품질 일관성 확보
- 고객 안내 누락 감소
- Outlook 사용 환경에 최적화된 이메일 생성
- 향후 자동화 프로젝트와의 연계 기반 마련

---

# 14. 향후 발전 방향

현재 프로젝트는 브라우저 기반의 독립형 애플리케이션으로 구현되어 있지만, 장기적으로는 운송 진행 관리와 이메일 자동화를 연결하는 방향으로 확장할 계획입니다.

주요 확장 방향은 다음과 같습니다.

- Progress Sheets 데이터 자동 연계
- Outlook Draft 자동 생성
- Microsoft Graph API 연동
- Power Automate 연계
- n8n Workflow 연계
- 이메일 생성 이력 관리
- 새로운 이메일 템플릿 추가
- 다국어 지원 확대

이러한 확장을 통해 반복적인 이메일 작성뿐 아니라 운송 진행 상황에 따른 고객 커뮤니케이션까지 하나의 워크플로우로 연결하는 것을 목표로 하고 있습니다.

# 15. 개발 과정 요약

Email Draft Generator는 반복적인 이메일 작성 업무를 줄이기 위한 작은 내부 도구에서 시작했습니다.

실제 업무에서 반복적으로 발생하는 문제를 하나씩 개선하는 과정을 거치면서, 단순한 이메일 생성 기능에서 조건부 템플릿, 실시간 Preview, Outlook 호환 복사, 다국어 지원을 갖춘 현재의 구조로 발전했습니다.

프로젝트를 진행하면서 가장 중요하게 고려한 것은 새로운 기능을 많이 추가하는 것이 아니라, 실제 업무에서 반복되는 작업을 얼마나 자연스럽게 줄일 수 있는가였습니다.

또한 현재 프로젝트를 독립적인 웹 애플리케이션으로 완성하는 것에 그치지 않고, Progress Sheets 자동화 및 Outlook Draft 자동화와 연계하여 운송 진행 관리와 고객 커뮤니케이션을 하나의 자동화 워크플로우로 연결할 수 있는 기반을 마련했습니다.

이러한 개발 과정을 통해 현재 프로젝트는 반복 업무를 줄이는 생산성 도구이면서, 향후 운송 자동화 생태계로 확장 가능한 이메일 생성 엔진으로 발전할 수 있는 구조를 갖추게 되었습니다.
